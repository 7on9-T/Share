IIS身份验证类型
-
### 1.	匿名身份验证  
原理：允许任何用户访问网站资源而无需提供凭据，IIS 会为匿名用户创建一个默认的用户账户（通常是 IIS_IUSRS 组中的一个账户）来代表他们进行访问操作。

适用场景：适用于公开的、不需要限制访问权限的信息页面。
### 2.	基本身份验证  
原理：用户通过浏览器输入用户名和密码 IIS 接收后会对照其存储的用户账户信息进行验证。

适用场景：在一些内网环境中，对安全性要求不是极高且需要简单的用户身份区分的场景下可使用，不过由于其传输的安全性问题，往往配合使用 SSL/TLS 加密通道来保障传输过程中用户名和密码的安全。
### 3.	摘要式身份验证  
原理：它比基本身份验证更安全一些，会使用hash对用户凭据进行处理后再在网络上传输，服务器端收到后同样进行相应的验证操作，且验证过程中结合了服务器上的一个特定密钥来增加安全性，这样避免了像基本身份验证那样容易被直接解码获取原始凭据的风险。

适用场景：适合在有一定安全需求的网络环境中使用，尤其是在域环境下，能较好地利用活动目录的用户账户信息进行验证，保障网站资源被合理授权访问。
### 4.	Windows 身份验证  
原理：依靠 Windows 操作系统的用户账户和安全机制来进行身份验证。用户在访问网站时，如果当前登录的 Windows 账户具备相应权限，IIS 会直接使用该账户信息进行验证，无需额外输入用户名和密码（在集成 Windows 身份验证等模式下，是透明化的验证过程）；如果没有权限则会提示相应的错误或者要求提供有权限的账户信息。

适用场景：常用于企业内部的基于 Windows 域的网络环境，方便利用已有的域用户账户体系来管理网站、应用程序的访问权限，比如企业内部的办公系统、业务管理系统等，保障只有内部授权员工可以访问。
### 5.	Forms 身份验证  
原理：这是一种基于表单的自定义身份验证方式。开发人员可以自行设计登录表单页面，用户在表单中输入用户名和密码后，由应用程序代码在后端按照自定义的逻辑（通常是与数据库等存储用户信息的地方进行交互验证）来验证身份，验证成功后应用程序会管理用户的登录状态等信息。

适用场景：在开发ASP.NET等类型的 Web 应用程序时广泛使用，适合创建个性化的用户登录和验证体系，例如常见的电商网站、社交平台等需要用户注册并登录后使用诸多个性化功能的应用场景。  

MIME处理类型  
-
### 1.	客户端识别与处理：
在网络传输过程中，客户端（比如浏览器、邮件客户端等）接收到数据时，通过 MIME 类型来判断接收到的数据是什么类型，进而采取相应的处理方式。例如浏览器收到 text/html 类型的数据就会进行网页渲染，收到 image/jpeg 类型的数据就会显示对应的图片。
### 2.	服务器配置与响应：
服务器端在发送数据时，需要正确设置相应的 MIME 类型，以便客户端能准确识别和处  理。同时，在服务器配置文件（如 Web 服务器的配置文件）中也可以基于 MIME 类型来设置不同的处理规则，比如对特定类型文件的缓存策略、访问权限等。
### 3.	确保兼容性与正确交互：
不同的应用程序、操作系统等之间通过遵循统一的 MIME 类型标准，能够保证数据在网络上的准确传递和交互，避免因无法识别数据类型而出现处理错误的情况，比如确保音频文件能被正确的音频播放软件识别并播放，视频文件能被对应的视频播放器正常播放等。

站库分离
-
站库分离指的是将网站应用程序（通常包括网页文件、脚本文件等前端相关内容）与数据库系统分开部署在不同的服务器或者不同的存储环境中。这样做的目的在于通过物理隔离提升系统整体的安全性、性能以及可维护性等。

辨别：子域名模式、端口模式、目录模式
-
### 1.子域名模式
子域名模式是通过不同的子域名来访问不同的应用或服务。例如：
```
app1.example.com
app2.example.com
admin.example.com
```
#### 特点：
每个子域名可以指向独立的服务器或服务。  
不同子域名之间的应用完全独立，适合多租户场景。  
子域名可以被搜索引擎视为独立站点。  
需要为每个子域名配置DNS记录。  
#### 适用场景：
多租户系统（如SaaS平台）。  
大型网站的不同功能模块（如博客、商店、论坛）。  
独立的环境（如开发、测试、生产环境）。  
### 2.端口模式
端口模式是通过不同的端口号来访问不同的应用或服务。例如：
```
example.com:8080
example.com:8888
example.com:7001
```
#### 特点
无需配置DNS或虚拟主机，直接通过端口区分。  
可以在同一服务器上运行多个服务。  
用户需要记住端口号，体验较差。  
开放多个端口可能增加安全风险。  
#### 适用场景：
- 开发和测试环境。
- 内部系统或工具（如数据库管理、监控工具）。
- 小型项目或临时服务。
### 3.目录模式
目录模式是通过URL路径来区分不同的应用或服务。例如：
```
example.com/app1
example.com/app2
example.com/admin
```
#### 特点：
只需在服务器上配置路由规则。  
URL结构清晰，易于理解。  
所有应用共享同一个域名和资源。  
目录结构对搜索引擎优化有利。  
#### 适用场景：
- 小型网站或单域名下的多应用。
- 功能模块划分清晰的系统（如后台管理、用户中心）。
- 静态网站或内容管理系统（CMS）。

白盒、黑盒测试
-

### 白盒测试
白盒测试也称为**结构测试**或**透明盒测试**，测试人员需要了解程序的内部结构、代码逻辑和实现细节，基于代码的内部逻辑设计测试用例。

### 黑盒测试
黑盒测试也称为**功能测试**或**行为测试**，测试人员无需了解程序的内部结构，仅从用户角度测试系统的功能是否符合需求。

路由访问
-
Web路由是Web开发中的核心机制，它决定了用户请求如何被处理。通过定义路由规则，开发者可以将不同的URL路径映射到相应的处理程序，从而实现Web应用的功能。不同的Web框架有不同的路由实现方式，但基本概念和工作流程是相似的。掌握路由的使用是构建Web应用的基础。

### Web路由的核心概念
#### URL路径（URL：统一资源定位器）：
用户访问的地址，例如 /home、/about、/user/123 等。
URL路径可以包含静态部分（如 /about）和动态部分（如 /user/123）。
#### 路由规则：
定义了URL路径与处理程序之间的映射关系。
例如，/home 可能映射到 HomeController 的 index 方法。
#### 处理程序（Handler）：
处理程序是实际处理请求的代码，通常是一个函数或方法。
处理程序负责生成响应内容并返回给用户。
#### HTTP方法：
路由通常与HTTP方法（如 GET、POST、PUT、DELETE 等）结合使用。
例如，GET /users 和 POST /users 可以分别映射到不同的处理程序。

### Web路由的工作流程
#### 用户请求：
用户在浏览器中输入URL或点击链接，发起HTTP请求。
#### 路由匹配：
Web框架根据URL路径和HTTP方法匹配路由规则，找到对应的处理程序。
#### 处理请求：
处理程序执行相应的逻辑，生成响应内容。
#### 返回响应：
服务器将响应内容返回给用户。

### Web路由的类型
#### 静态路由：
URL路径是固定的，例如 /home、/about。
适用于不需要动态参数的场景。
#### 动态路由：
URL路径包含动态部分，例如 /user/123、/post/2023/10/05。
动态部分通常作为参数传递给处理程序。
#### 通配符路由：
使用通配符匹配多个URL路径，例如 /files/* 可以匹配 /files/a、/files/b/c 等。
适用于需要匹配一组URL的场景。
#### 正则表达式路由：
使用正则表达式定义复杂的路由规则。
例如，/user/[0-9]+ 可以匹配 /user/123，但不匹配 /user/abc。
